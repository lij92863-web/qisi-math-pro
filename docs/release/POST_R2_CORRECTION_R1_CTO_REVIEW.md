# Post-R2 Correction R1 CTO Review

POST_R2_CORRECTION_ACCEPTED_WITH_LIMITATIONS

## Production-wired

- Formal Admission Policy is the unified batch formal-entry policy.
- The storage repository transaction is the formal question/image/draft/batch
  write owner, reached through batch formal submit.
- The production review validator composes schema, manual revision,
  provenance, and admission precheck and is wired into the app controller.
- Controlled-write remains the PDF/AI answer and solution ownership boundary.
- Deterministic DOCX and safe-partial PDF paths reach review persistence and
  formal admission/rejection through production-linked browser tests.
- Archive security, temporary-job cleanup, private-safe logging, library,
  export, support parsing/alignment, and the honest legacy batch coordinator are
  production-wired.

## Scaffold

- Qwen OCR adapter, local OCR adapter, OCR registry, shadow mode, and the
  performance monitor remain scaffold components unless separately promoted.
- The aspirational ImportOrchestrator remains scaffold; production truth is the
  explicitly named LegacyBatchRunCoordinator plus the marked deterministic
  injected path used by true E2E.
- Route B is not scaffold or production: it remains research-only and frozen.

## Real OCR not measured

- No real AI/OCR request or model download was authorized in Program A.
- Deterministic candidates prove control flow, ownership, persistence, reload,
  and export; they do not prove OCR accuracy, CER, formula fidelity, or teacher
  correction cost on private materials.
- Therefore no OCR engine is production-promoted by this review.

## Program B

- Measure approved real corpora and compare engine accuracy, formula fidelity,
  structure/options, wrong attachment, latency, cost, and manual correction.
- Decide which engine, if any, is promoted; keep non-winning engines in shadow
  or research-only status without weakening controlled-write.

## Program C

- Reduce the 21,777-line app shell and decompose `processDraftImportBatch`
  without duplicating owners or moving unsafe writes into new files.
- Address six explicitly inventoried non-batch direct question-write sites,
  diagnostic globals, and broad fallback orchestration.
- Re-benchmark coupling, maintainability, browser performance, and ownership
  after each bounded migration package.

The correction objective is accepted because the formal data boundary,
fail-closed controllers, transaction/idempotency behavior, deterministic true
E2E, security attacks, and benchmark gate all passed. The limitations above are
material and are entry conditions for Programs B and C, not claims completed by
Program A.
